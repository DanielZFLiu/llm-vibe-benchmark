# System Design Document: "Secret Society"

## 1. Architecture Overview

### 1.1 High-Level Design
"Secret Society" is designed as a monolithic application containerized for simplicity, running on a single VPS. The architecture prioritizes data isolation and cryptographic integrity over distributed scalability.

**Components:**
1.  **Client (Web/Mobile):** React/TypeScript (Web) or Swift/Kotlin (Mobile). Handles all cryptographic operations locally.
2.  **API Server (Go):** Single binary handling HTTP REST, WebSocket connections, and business logic. Go is chosen for static compilation, low memory footprint, and strong concurrency primitives.
3.  **Database (PostgreSQL):** Stores user metadata, message envelopes (ciphertext), and invite logs.
4.  **Blob Storage (Local Disk):** Encrypted media files stored on the VPS filesystem, referenced by DB.
5.  **Proxy (Caddy):** Handles TLS termination and automatic certificate renewal.

### 1.2 Component Diagram

```text
       [ Client ]                         [ Client ]
          |                                  |
          | (WebSocket + HTTPS)              |
          v                                  v
       +-----------------------------------------+
       |           Caddy (TLS Proxy)             |
       +-----------------------------------------+
          |                                  |
          v                                  v
       +-----------------------------------------+
       |          Secret Society API (Go)        |
       |  - Auth Logic  - Invite Logic           |
       |  - WS Router   - Crypto Metadata        |
       +-----------------------------------------+
          |                                  |
          v                                  v
       +------------------+    +------------------+
       |   PostgreSQL     |    |   Local Storage  |
       | (Metadata +      |    | (Encrypted Media)|
       |  Ciphertext)     |    |                  |
       +------------------+    +------------------+
```

### 1.3 Justification of Choices
*   **Monolithic Go Binary:** Reduces operational complexity. No need to orchestrate multiple microservices on a single VPS.
*   **PostgreSQL:** Robust, easy to backup (`pg_dump'), and supports JSONB for flexible metadata without schema migrations for every client update.
*   **WebSockets:** Essential for real-time presence and typing indicators without the overhead of polling.
*   **Local Storage:** For 1,000 users, S3 is unnecessary cost/complexity. Local disk is faster and easier to backup via standard VPS tools.

## 2. End-to-End Encryption (E2EE) Design

This system uses a hybrid approach: **Double Ratchet** for 1:1 chats and **Sender Key Distribution** for groups, built on **libsodium** primitives.

### 2.1 Key Generation & Identity
*   **Identity Key Pair (`IdK`):** Generated on client signup.
    *   *Private:* Stored only on device (encrypted by local passphrase).
    *   *Public:* Uploaded to server (`users.public_key`). Used to identify the user and encrypt initial handshakes.
*   **Device Key Pair:** If a user has multiple devices, each generates a unique device key, linked to the Identity Key.

### 2.2 Conversation Initiation (1:1)
1.  **Pre-Keys:** Clients upload a list of one-time public keys (`PreK`) to the server.
2.  **Handshake:** Sender A fetches Receiver B's `PreK` from server.
3.  **Session Init:** A performs X25519 ECDH with B's `PreK` and `IdK` to generate a shared secret.
4.  **Double Ratchet:** This shared secret initializes the Ratchet state. Every message increments the ratchet, providing **Forward Secrecy**.
5.  **Server Role:** Server routes the ciphertext. It sees the `session_id` metadata but cannot derive the key.

### 2.3 Group Chat (Up to 50 Members)
Implementing Double Ratchet for 50 users is computationally expensive. We use the **Sender Key** pattern (similar to Signal/WhatsApp).
1.  **Group Creation:** Admin creates a Group Room.
2.  **Group Sender Key (`GSK`):** A random AES-256 key generated by the creator.
3.  **Distribution:** The `GSK` is encrypted individually using each member's 1:1 E2EE session and sent to them.
4.  **Messaging:** Members encrypt messages with `GSK`. All members can decrypt.
5.  **Rotation:** To maintain forward secrecy in groups, the `GSK` is rotated periodically (e.g., every 100 messages or when a member leaves). The new `GSK` is distributed via 1:1 E2EE channels.

### 2.4 Forward Secrecy & Compromise
*   **Mechanism:** The Double Ratchet ensures that if a current state key is compromised, *past* messages cannot be decrypted (because the ratchet state was overwritten).
*   **Long-Term Key Compromise:** If a user's `IdK` private key is stolen, an attacker can impersonate them and read *future* group keys distributed to them. It does **not** allow reading past 1:1 message history (due to ratchet), but does allow reading future group history.
*   **Mitigation:** Users must regenerate Identity Keys and re-verify sessions if compromise is suspected.

### 2.5 Storage & Retrieval
*   **Server Storage:** Stores `ciphertext` (blob), `sender_id`, `room_id`, `timestamp`, `nonce`.
*   **Client Retrieval:** Client polls/fetches messages via WebSocket. Client downloads the blob, looks up the local session state, decrypts, and renders.
*   **Ephemeral Messages:** The server runs a nightly job `DELETE messages WHERE expiry < NOW()`. The client also locally hides them immediately upon decryption.

### 2.6 Key Restoration (New Device)
True E2EE means the server cannot reset keys.
*   **Recovery Kit:** Upon signup, the client generates a `Recovery Code` (password-encrypted bundle of private keys).
*   **Storage:** User must save this locally (print it, save to USB).
*   **Optional Server Vault:** We offer an *optional* "Key Vault" feature. The client encrypts the private keys with a **User Password** (never sent to server) and uploads the blob to the server.
    *   *Trade-off:* If the user forgets the password, data is lost. If the server is compromised, the blob is useless without the password.

## 3. Invite System

The invite system is the gatekeeper. It must be auditable but privacy-preserving regarding the *target* until redemption.

### 3.1 Invite Generation
*   **Action:** Member A clicks "Generate Invite".
*   **Token:** Server generates a cryptographically secure random string (e.g., 32-byte hex), e.g., `INV-<hash>`.
*   **Record:** Server stores `invites` table: `{ token, inviter_id (A), created_at, used_by_id (NULL), status (ACTIVE) }`.
*   **Constraint:** Server knows A generated an invite, but does not know *who* will use it.

### 3.2 Validation & Redemption
*   **Action:** Outsider B lands on signup page, enters `INV-<hash>` and their details.
*   **Validation:** Server checks if token exists and is `ACTIVE`.
*   **Linking:** Upon successful registration, Server updates `invites`: `used_by_id = B`.
*   **Privacy:** The server only links A to B *after* B has successfully authenticated and joined. The token itself does not contain B's identity.

### 3.3 Auditability (Invite Tree)
*   **Admin View:** A graph view showing `A -> B -> C`.
*   **Query:** Recursive SQL query on `invites` table to trace lineage.
*   **Immutable Log:** Once `used_by_id` is set, it cannot be changed. This creates a permanent audit trail of trust propagation.

### 3.4 Revocation & Chain Pruning
*   **Revoking an Invite:** Admin marks token as `REVOKED`. If B hasn't joined yet, B cannot join.
*   **Revoking a Member:** If Member A is banned:
    1.  A's account is disabled.
    2.  All *unused* invites generated by A are auto-revoked.
    3.  **The Chain:** Existing members invited by A (e.g., B) remain active *unless* the Society policy is "Trust Tree Pruning."
    4.  **Design Choice:** We implement **Soft Pruning**. B remains a member but cannot generate new invites. This prevents mass excommunication due to one bad actor, while stopping the spread of their trust.

### 3.5 Abuse Prevention
*   **Rate Limiting:** Max 5 active invites per member at any time.
*   **Expiration:** Invites expire after 7 days automatically.
*   **One-Time Use:** Token is invalidated immediately after first successful redemption.
*   **Replay Attack:** Tokens are single-use; attempting to use a redeemed token fails validation.

## 4. Data Model

The database separates **Routing Metadata** (plaintext, needed for server logic) from **Content** (ciphertext, opaque to server).

### 4.1 Core Tables (PostgreSQL)

```sql
-- Users: Identity and Auth
CREATE TABLE users (
    id UUID PRIMARY KEY,
    username TEXT UNIQUE,          -- Plaintext (Directory)
    public_key BYTEA,              -- Plaintext (E2EE Identity)
    password_hash TEXT,            -- Plaintext (Login Auth only)
    is_admin BOOLEAN,
    status TEXT,                   -- 'active', 'banned', 'revoked'
    created_at TIMESTAMP
);

-- Invites: Audit Trail
CREATE TABLE invites (
    token TEXT PRIMARY KEY,
    inviter_id UUID REFERENCES users(id),
    redeemed_by_id UUID REFERENCES users(id), -- NULL until joined
    status TEXT,                   -- 'active', 'redeemed', 'revoked', 'expired'
    created_at TIMESTAMP,
    expires_at TIMESTAMP
);

-- Rooms: Chat Context
CREATE TABLE rooms (
    id UUID PRIMARY KEY,
    type TEXT,                     -- 'direct', 'group'
    creator_id UUID REFERENCES users(id),
    ephemeral_ttl INTERVAL         -- NULL = permanent, e.g., '24 hours'
);

-- Room Members: Access Control
CREATE TABLE room_members (
    room_id UUID REFERENCES rooms(id),
    user_id UUID REFERENCES users(id),
    joined_at TIMESTAMP,
    PRIMARY KEY (room_id, user_id)
);

-- Messages: The Envelope
CREATE TABLE messages (
    id UUID PRIMARY KEY,
    room_id UUID REFERENCES rooms(id),
    sender_id UUID REFERENCES users(id),
    ciphertext BYTEA,              -- OPAQUE (E2EE Content)
    nonce BYTEA,                   -- Required for decryption
    timestamp TIMESTAMP,
    delivered BOOLEAN              -- For read receipts
);

-- Keys: Public Routing Info
CREATE TABLE user_keys (
    user_id UUID REFERENCES users(id),
    key_type TEXT,                 -- 'identity', 'prekey', 'device'
    key_data BYTEA,                -- Public key blob
    uploaded_at TIMESTAMP
);
```

### 4.2 Encryption Boundaries
*   **Plaintext:** `username`, `public_key`, `room_id`, `sender_id`, `timestamp`. The server needs this to route messages and render the directory.
*   **Ciphertext:** `message content`, `file attachments`, `private keys` (if stored in vault).
*   **Metadata Privacy:** The server *knows* A sent a message to Room B at Time T. It does not know *what* the message said.

## 5. Real-Time Protocol

### 5.1 Transport: WebSockets
*   **Why:** Persistent connection allows immediate push of messages, typing indicators, and presence updates without polling overhead.
*   **Connection:** Client opens `wss://domain/ws`. Authenticated via JWT (short-lived) exchanged during HTTPS login.

### 5.2 Message Routing
1.  **Send:** Client A encrypts payload -> Sends WS packet `{ room_id, ciphertext, nonce }`.
2.  **Server:** Validates session, inserts to DB, looks up `room_members` for `room_id`.
3.  **Push:** Server iterates active WebSocket connections for members in `room_members` and pushes the packet.
4.  **Receipt:** Client B receives packet, decrypts locally, sends ACK `{ message_id, read_at }` back to server. Server updates `messages.delivered`.

### 5.3 Presence & Typing
*   **Presence:** Client sends heartbeat every 30s. If WS disconnects, server sets `last_seen = NOW()`.
*   **Typing:** Client sends `typing: true` event. Server broadcasts to room members. Expires automatically after 5s of inactivity to avoid state leakage.

### 5.4 Offline Delivery
*   **Queue:** Messages are persisted in DB immediately.
*   **Sync:** When Client reconnects, it requests `GET /messages?after=<last_id>`.
*   **State:** Server tracks `last_sync_id` per user. On login, server returns count of unread messages.

## 6. Self-Hosting & Operations

### 6.1 Deployment Stack
The entire system is defined in a single `docker-compose.yml`.

```yaml
version: '3'
services:
  caddy:
    image: caddy:alpine
    volumes: ./caddy_data:/data
  api:
    image: secret-society-api:latest
    depends_on: [db]
    environment: DB_HOST=db
  db:
    image: postgres:15
    volumes: ./pg_data:/var/lib/postgresql
```

### 6.2 External Dependencies
*   **DNS:** Point domain to VPS IP.
*   **TLS:** Caddy auto-provisions via Let's Encrypt.
*   **SMTP:** Optional. If email invites are used, configure SMTP creds in env. Otherwise, invites are shared manually (copy/paste token).
*   **Storage:** Media stored in `/app/uploads` inside the container, mapped to host volume.

### 6.3 Backup & Restore
*   **Database:** Daily cron job on VPS: `pg_dump | gzip > backup.sql.gz`.
*   **Encryption:** Backups should be encrypted (e.g., using `age`) before storage, as DB contains invite trees and public keys.
*   **Key Material:** **Critical:** User private keys are *not* in the DB (unless using the optional Key Vault). Backing up the DB does not allow restoring user chat history. Users must manage their own key backups.
*   **Restore:** Spin up new VPS, restore volume, run `docker-compose up`.

### 6.4 Monitoring
*   **Health Check:** `/api/health` endpoint returns 200 if DB connected.
*   **Logs:** Structured JSON logs to stdout. VPS log driver rotates files.
*   **Alerts:** Simple shell script monitoring disk space and container restart counts. No complex Prometheus stack needed for 1k users.

## 7. Threat Model

### 7.1 Malicious Server Operator
*   **Can Observe:**
    *   Who is talking to whom (Social Graph).
    *   When they are talking (Traffic Patterns).
    *   Message size (approximate content length).
    *   Invite tree (Trust hierarchy).
    *   Usernames and Public Keys.
*   **Cannot Observe:**
    *   Message content (Text, Files).
    *   Group Keys (unless they actively MITM the key distribution, which requires client compromise).
    *   User Passwords (hashed).

### 7.2 Database Exfiltration
*   **Impact:** Attacker gets the `messages` table.
*   **Result:** Data is useless. `ciphertext` cannot be decrypted without client-side private keys stored on user devices.
*   **Metadata Leak:** The attacker learns the social structure of the society (who invited whom, who talks to whom). This is the primary privacy risk.

### 7.3 Device Compromise
*   **Blast Radius:**
    *   If Device A is rooted/hacked: All messages *on Device A* are readable.
    *   Future Messages: Attacker can extract the Session State and read future messages in that specific 1:1 channel until the ratchet rotates or session is reset.
    *   Group Chat: Attacker can extract the `Group Sender Key` and read all group traffic.
*   **Mitigation:** Remote Wipe is impossible (E2EE). User must notify Admin to revoke their membership, which invalidates their ability to receive *new* group keys, but cannot revoke old keys already distributed.

### 7.4 Invite System Attacks
*   **Token Guessing:** Mitigated by 256-bit entropy tokens.
*   **Invite Theft:** If a token is intercepted (e.g., sent via insecure channel), an attacker can join.
    *   *Defense:* Tokens should be shared via existing E2EE channels or physically.
*   **Chain Poisoning:** If a bad actor joins and invites others.
    *   *Defense:* Admin audit tools allow tracing the tree. Admins can bulk-revoke descendants of a bad actor.

### 7.5 Explicit Trade-offs
1.  **Metadata vs. Complexity:** We chose to store `sender_id` and `room_id` in plaintext. A zero-metadata design (like some research prototypes) would require fully decentralized routing, which breaks the "Self-Hosted Single VPS" constraint. We accept metadata leakage to ensure operational feasibility.
2.  **Key Recovery vs. Security:** We offer an optional password-encrypted Key Vault. This risks password guessing attacks but prevents users from losing access. The default design (user manages keys) is more secure but less usable.
3.  **Group Scale:** We limit groups to 50. Scaling E2EE groups to 1,000 users requires advanced cryptographic engineering (e.g., MLS) which is beyond the scope of a single-VPS deployment team.

---

## Conclusion
"Secret Society" balances strong cryptographic guarantees with the operational reality of a small, self-hosted community. By leveraging standard primitives (libsodium), a robust relational schema, and a strict invite audit trail, it provides a secure environment for trusted communication without requiring enterprise-scale infrastructure. The design acknowledges that while content is secret, the social graph is visible to the operator—a necessary trade-off for right-sized architecture.
